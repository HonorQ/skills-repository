# Error Solver

系统性解决代码中的错误、异常和 Bug，从定位到修复的完整流程。

## 核心原则

### 1. 先定位，后修复
不急于给解决方案，而是先理解问题本质：

```
❌ 错误做法：看到报错就给出修复代码
✅ 正确做法：先分析错误原因，再给出针对性方案
```

### 2. 穷尽现有方案
修复前先检查：
- 是否已有类似错误处理
- 是否有现有的工具函数可用
- 是否符合项目现有模式

### 3. 最小化改动
优先选择影响范围小的修复方案：

| 优先级 | 方案类型 | 示例 |
|--------|----------|------|
| 1 | 配置/参数调整 | 修改配置项、传入正确参数 |
| 2 | 局部代码修复 | 修改单行或单个函数 |
| 3 | 增加容错处理 | 添加判空、try-catch |
| 4 | 结构调整 | 修改调用顺序、重构逻辑 |
| 5 | 架构改动 | 最后的选择 |

## 错误分类与处理策略

### 类型 1：编译/语法错误

**特征**：代码无法运行，编辑器标红

**处理流程**：
```
1. 定位错误位置（行号、文件）
2. 理解错误信息含义
3. 检查语法规则
4. 提供修复方案
5. 验证修复后无新错误
```

**常见场景**：
- 缺少括号/引号
- 类型不匹配
- 未定义变量
- 导入错误

### 类型 2：运行时错误

**特征**：代码能运行但中途崩溃

**处理流程**：
```
1. 分析堆栈跟踪（stack trace）
2. 定位崩溃位置
3. 分析触发条件
4. 添加防御性代码
5. 验证边界情况
```

**常见场景**：
- 空指针/undefined 访问
- 数组越界
- 类型转换错误
- 异步处理错误

### 类型 3：逻辑错误

**特征**：代码运行但结果不对

**处理流程**：
```
1. 明确预期结果 vs 实际结果
2. 添加日志/断点追踪
3. 定位逻辑偏差
4. 修正算法/条件
5. 补充测试用例
```

**常见场景**：
- 条件判断错误
- 计算逻辑错误
- 状态管理错误
- 时序问题

### 类型 4：测试失败

**特征**：测试用例不通过

**处理流程**：
```
1. 分析失败断言
2. 判断是代码问题还是测试问题
3. 定位差异来源
4. 修复代码或更新测试
5. 确保所有测试通过
```

### 类型 5：性能问题

**特征**：运行缓慢、内存泄漏、卡顿

**处理流程**：
```
1. 性能分析定位瓶颈
2. 分析算法复杂度
3. 优化关键路径
4. 验证性能提升
5. 确保功能正确
```

## 调试流程

### Step 1: 收集信息

询问或提取以下信息：

```
📋 错误信息
- 完整的错误提示
- 堆栈跟踪（stack trace）
- 错误发生的文件和行号

🔄 复现步骤
- 什么操作触发了错误
- 是否每次都发生
- 什么情况下不发生

🌍 环境信息
- 开发/测试/生产环境
- 相关配置
- 最近是否有改动

📦 相关代码
- 出错的代码片段
- 相关的调用链
- 最近修改的代码
```

### Step 2: 分析定位

使用以下方法定位问题：

**二分法定位**
```
在关键位置添加日志/断点：
1. 确认数据流入是否正确
2. 确认处理过程是否正确
3. 确认数据流出是否正确
```

**对比法**
```
对比正常情况和异常情况：
- 参数差异
- 环境差异
- 数据差异
```

**假设验证**
```
提出可能原因 → 设计验证方案 → 确认/排除
```

### Step 3: 根因分析

找到根本原因，而非表面现象：

```
表面现象：报错 "Cannot read property 'x' of undefined"

直接原因：某个对象是 undefined

根因分析：
- 为什么会是 undefined？
- 数据从哪里来？
- 哪个环节没有处理好？
- 是预期内还是预期外？

根本原因：API 返回数据格式变化，未做兼容处理
```

### Step 4: 制定方案

提供多个修复方案供选择：

```
## 修复方案

### 方案 A：快速修复（推荐）
**改动**：添加判空处理
**优点**：改动小，风险低
**缺点**：未解决根本问题
**适用**：紧急修复

### 方案 B：根本修复
**改动**：统一数据校验层
**优点**：解决根本问题，防止类似错误
**缺点**：改动范围大，需要测试
**适用**：非紧急，有测试时间

### 方案 C：防御性编程
**改动**：增加类型检查 + 错误提示
**优点**：增强健壮性，便于排查
**缺点**：代码量增加
**适用**：公共接口、核心模块
```

### Step 5: 实施修复

按规范实施修复：

```
1. 先备份或确认可回滚
2. 实施选定的修复方案
3. 验证修复有效
4. 检查是否有副作用
5. 补充测试用例
```

### Step 6: 验证与预防

```
✅ 验证清单
- [ ] 原错误不再出现
- [ ] 相关功能正常
- [ ] 边界情况测试通过
- [ ] 无新错误引入

🛡️ 预防措施
- [ ] 添加测试覆盖
- [ ] 更新文档/注释
- [ ] 代码审查
- [ ] 监控告警（生产环境）
```

## 常见错误速查

### JavaScript/TypeScript

| 错误 | 可能原因 | 解决方案 |
|------|----------|----------|
| Cannot read property 'x' of undefined | 未判空访问嵌套属性 | 可选链 `?.` 或判空 |
| x is not a function | 类型错误，调用了非函数 | 检查类型，确认导入正确 |
| Cannot find module 'x' | 模块未安装或路径错误 | npm install / 检查路径 |
| Promise 未处理 rejection | async/await 缺少 try-catch | 添加错误处理 |
| 内存泄漏 | 事件监听未移除、闭包引用 | 清理监听器、解除引用 |

### Python

| 错误 | 可能原因 | 解决方案 |
|------|----------|----------|
| KeyError | 字典访问不存在的 key | 用 .get() 或判存 |
| IndexError | 列表越界访问 | 检查长度后再访问 |
| AttributeError | 对象没有该属性 | 检查对象类型 |
| ImportError | 模块导入失败 | 检查安装和路径 |
| TypeError | 类型不匹配 | 检查参数类型 |

### 通用问题

| 问题 | 排查方向 |
|------|----------|
| 偶现 Bug | 时序、竞态条件、异步问题 |
| 只在某环境出现 | 配置差异、数据差异、版本差异 |
| 升级后出现 |  breaking change、依赖冲突 |
| 本地正常线上异常 | 环境差异、数据量差异、构建差异 |

## 最佳实践

1. **保持冷静**：错误是正常现象，系统性分析比快速尝试更有效
2. **复现第一**：无法复现的错误难以修复
3. **最小改动**：每次只改一处，方便定位
4. **版本控制**：修复前确保可以回滚
5. **测试验证**：修复后必须验证，包括边界情况
6. **文档记录**：复杂错误记录解决方案，便于后续参考
7. **预防为主**：修复后思考如何避免类似错误
