# Analyze First

AI 先分析需求，提供多种实现方案，用户选择后再执行。

## 快速入口

**用户说** → **AI 做**

| 用户意图 | 触发词示例 | AI 响应 |
|----------|------------|---------|
| 规划 | "先分析一下"、"plan first" | 分析需求，提供方案 |
| 对比 | "有什么方案"、"compare approaches" | 对比多种实现方式 |
| 犹豫 | "不确定用哪种"、"help me decide" | 推荐最佳方案 |
| 优化 | "有没有更好的方式"、"suggest alternatives" | 寻找改进空间 |

## 何时使用

### ✅ 适合使用

- 功能实现有多种方式
- 架构决策需要权衡
- 不确定最佳实践
- 想要避免返工

### ❌ 不适合使用

- 问题已经很明确，不需要分析
- 时间紧迫，需要立即行动
- 问题太小，不值得分析（直接用 `karpathy-guidelines`）
- 需要创意探索（用 `brainstorming`）

### vs 其他 Skill

| Skill | 用途 | 场景 |
|-------|------|------|
| **analyze-first** | 工程决策 | "怎么实现 X" |
| **brainstorming** | 创意探索 | "能用 X 做什么" |
| **writing-plans** | 执行计划 | 选定方案后的详细步骤 |

## 核心流程

### Step 0: 代码库分析

在提出方案前，先了解现有代码：
- 检查 package.json、tsconfig.json 等配置
- 识别技术栈和框架
- 搜索已有的类似实现
- 理解代码规范和目录结构

### Step 1: 需求分析

- 理解核心目标
- 识别约束条件
- 澄清模糊之处
- 判断复杂度级别

### Step 2: 深度控制

| 深度 | 场景 | 方案数 | 详情 |
|------|------|--------|------|
| **快速** | 简单决策 | 2-3 | 名称 + 优缺点 |
| **标准** | 功能决策 | 3-4 | + 复杂度 + 工作量 |
| **深度** | 架构决策 | 3-5 | + 风险 + 回退难度 |

用户可指定："快速分析一下" / "详细分析" / "deep dive"

### Step 3: 方案生成

按推荐度排序，每个方案包含：
- 方案名称和描述
- 优点和缺点
- 复杂度和工作量 *(标准+)*
- 风险和回退难度 *(深度)*
- 新依赖 *(深度)*

### Step 4: 推荐与置信度

**置信度级别**：
- **强推荐**: 最佳选择，几乎没有争议
- **建议**: 好的选择，取决于优先级
- **仅供参考**: 多个可行选项，取决于上下文

**必须说明**：为什么不推荐其他方案

### Step 5: 用户选择

**动态菜单规则**：根据分析结果中有几个方案，动态生成选择菜单：

**场景 A: 仅有单一强推荐（无备选方案）**
当分析结果只有一个推荐方案，且没有其他备选方案时，使用简化菜单：

```
请选择:
1. 采用此方案
2. 需要更多信息
3. 有疑问想讨论
```

**场景 B: 有多个方案（含备选）**
当分析结果有推荐方案和备选方案时，使用完整菜单：

```
请选择:
1. 使用推荐方案
2. 选择备选方案
3. 需要更多信息
4. 对某个方案有疑问
5. 快速验证 - 先做 POC
```

> 如用户明确要求"混合方案"或"渐进式方案"，再展开详细讨论。

> **判断标准**：如果输出中只展示了 1 个方案（只有 🥇 推荐，没有 🥈 备选或更多），则使用场景 A 的简化菜单。如果展示了 2 个及以上方案，使用场景 B 的完整菜单。

### Step 6: 可行性验证

建议如何验证方案：
- **快速 POC**: 最小化验证核心假设
- **试点实现**: 先在一个模块应用
- **技术调研**: 验证技术可行性
- **基准测试**: 测量性能影响

### Step 7: 决策记录

记录最终决策，方便后续回顾：
- 选择了哪个方案
- 为什么选择
- 放弃了哪些方案
- 已知风险和回退计划

## 项目规范关联

分析方案时，必须参考项目开发规范：
- **技术栈约束**: 项目强制使用 JavaScript，禁止 TypeScript
- **样式方案**: 优先使用 Tailwind CSS，避免覆盖 Element Plus 类
- **工具库**: 时间处理用 moment.js，日志用全局 logger
- **组件规范**: EnhancedTable 已全局注册，禁止重复 import

> 详细规范见 `.trae/rules/product-rules.md`

## 最佳实践

1. **先分析代码库**: 方案必须基于实际项目
2. **动态调整方案数**: 根据复杂度决定
3. **评估风险**: 帮助用户理解潜在问题
4. **提供灵活选择**: 支持混合、渐进、追问
5. **考虑用户偏好**: 根据偏好调整推荐
6. **给出可执行步骤**: 选择后提供明确指导
7. **标注置信度**: 帮助用户理解推荐强度
8. **解释不推荐**: 帮助用户理解权衡
9. **建议验证**: 帮助用户在全面实施前验证
10. **记录决策**: 帮助团队理解和回顾

## 与其他 Skill 集成

- **test-driven-development**: 实施前规划测试
