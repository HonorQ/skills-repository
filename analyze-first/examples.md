### 示例 5: 状态管理重构

**用户**: "我想重构状态管理，现在 Redux 太重了"

**AI 分析**:

```
## 代码库上下文
- 技术栈: React 18 + Redux Toolkit + React Query
- 相关代码: 30+ slice，大量样板代码，React Query 已处理服务端状态
- 项目约定: TypeScript strict mode

## 需求理解
- 核心目标: 简化状态管理，减少样板代码
- 关键约束: 渐进式迁移，不能一次性重写
- 假设条件: 服务端状态已由 React Query 处理

## 实现方案对比

| 排名 | 方案 | 描述 | 复杂度 | 工作量 | 风险 | 回退难度 | 新依赖 | 置信度 |
|------|------|------|--------|--------|------|----------|--------|--------|
| 🥇 | Zustand | 轻量级状态管理 | 简单 | 中等 | 低 | 简单 | zustand | 强推荐 |
| 🥈 | Jotai | 原子化状态管理 | 简单 | 中等 | 低 | 简单 | jotai | 建议 |
| 🥉 | 精简Redux | 保留Redux，减少slice | 中等 | 耗时 | 低 | 简单 | 无 | 建议 |
| 4 | ReactQuery+useState | 全部迁移到服务端状态 | 复杂 | 耗时 | 高 | 困难 | 无 | 仅供参考 |

## 推荐方案
**选择**: Zustand
**置信度**: 强推荐
**理由**:
1. API 简洁，样板代码少
2. 支持渐进式迁移，可与 Redux 共存
3. TypeScript 支持好
4. 社区活跃，被广泛采用

## 不推荐说明
- **Jotai**: 原子化模型适合细粒度状态，但对于全局状态管理不如 Zustand 直观
- **精简Redux**: 治标不治本，Redux 的样板代码问题是架构性的
- **ReactQuery+useState**: 过度依赖服务端状态，客户端状态需要专门的管理方案

## 可行性验证
**方式**: 模块试点
**时间**: 3-5 天
**验证步骤**:
1. 安装 zustand 依赖
2. 选择一个独立模块（如用户设置）
3. 用 Zustand 替换该模块的 Redux slice
4. 对比代码量和开发体验
5. 收集团队反馈

**验证指标**: 
- 代码量减少 30%+
- 开发体验提升（团队反馈）
- 无功能回归

## 决策记录
**日期**: 2026-05-29
**选择**: Zustand
**理由**: API 简洁，渐进式迁移友好，TypeScript 支持好
**放弃**: Jotai, 精简Redux, ReactQuery+useState
**放弃原因**: Jotai 不适合全局状态；精简Redux 治标不治本；ReactQuery+useState 过度依赖服务端状态

**假设条件**:
- 团队愿意学习新库
- 可以接受短期的混合状态管理
- 服务端状态已由 React Query 处理

**已知风险**:
- 团队学习成本
- 与 Redux DevTools 的兼容性
- 30+ slice 的迁移工作量

**回退计划**:
- 如果 Zustand 不适合，可以回退到精简 Redux
- 保留 Redux 代码，不删除
- 逐步迁移，不一次性切换

## 已知限制
- 没有评估团队对 Zustand 的学习成本
- 没有考虑与 Redux DevTools 的兼容性
- 没有评估 30+ slice 的迁移工作量
- 没有考虑与现有中间件的兼容性

## 下一步行动
- [ ] 用户确认方案选择
- [ ] 模块试点验证
- [ ] 制定迁移计划
- [ ] 逐步迁移其他模块
```

### 示例 6: API 架构

**用户**: "API 用 REST 还是 GraphQL？"

**AI 分析**:

```
## 代码库上下文
- 技术栈: Next.js 14 + TypeScript + Prisma + PostgreSQL
- 相关代码: 已有 50+ REST API 端点，前端使用 SWR 获取数据
- 项目约定: App Router, Server Actions

## 需求理解
- 核心目标: 优化 API 架构，提升开发效率和性能
- 关键约束: 已有 REST API 需要考虑迁移成本
- 假设条件: 需要支持复杂查询和实时更新

## 实现方案对比

| 排名 | 方案 | 描述 | 复杂度 | 工作量 | 风险 | 回退难度 | 新依赖 | 置信度 |
|------|------|------|--------|--------|------|----------|--------|--------|
| 🥇 | 保持REST+优化 | 优化现有REST API | 简单 | 中等 | 低 | 简单 | 无 | 建议 |
| 🥈 | REST+GraphQL混合 | 新功能用GraphQL | 中等 | 耗时 | 中 | 中等 | @apollo/client | 建议 |
| 🥉 | 全部迁移到GraphQL | 完全替换REST | 复杂 | 耗时 | 高 | 困难 | @apollo/client | 仅供参考 |

## 推荐方案
**选择**: 保持REST+优化
**置信度**: 建议
**理由**:
1. 已有 50+ REST API，迁移成本高
2. Next.js Server Actions 可以优化数据获取
3. SWR 已经处理了缓存和更新
4. 团队熟悉 REST，学习成本低

## 不推荐说明
- **REST+GraphQL混合**: 增加架构复杂度，需要维护两套 API
- **全部迁移到GraphQL**: 迁移成本高，风险大，收益不确定

## 可行性验证
**方式**: 技术调研
**时间**: 2-3 天
**验证步骤**:
1. 分析现有 API 的痛点
2. 评估 Server Actions 的适用场景
3. 测试 SWR 的高级功能
4. 评估性能提升空间

**验证指标**: 
- 明确优化方向
- 量化性能提升空间
- 评估迁移成本

## 决策记录
**日期**: 2026-05-29
**选择**: 保持REST+优化
**理由**: 迁移成本低，团队熟悉，Next.js 原生支持好
**放弃**: REST+GraphQL混合, 全部迁移到GraphQL
**放弃原因**: 增加复杂度；迁移成本高；收益不确定

**假设条件**:
- 现有 REST API 可以满足需求
- Server Actions 可以优化数据获取
- 团队愿意学习 SWR 高级功能

**已知风险**:
- 可能错过 GraphQL 的优势
- 复杂查询可能需要多次请求
- 实时更新需要额外处理

**回退计划**:
- 如果 REST 无法满足需求，可以引入 GraphQL
- 先在新功能试点 GraphQL
- 逐步迁移关键 API

## 已知限制
- 没有详细分析现有 API 的性能瓶颈
- 没有评估 Server Actions 的实际性能
- 没有考虑与第三方 API 的集成
- 没有评估实时更新的需求优先级

## 下一步行动
- [ ] 用户确认方案选择
- [ ] 分析现有 API 痛点
- [ ] 优化关键 API 性能
- [ ] 引入 Server Actions

```

## 总结

以上示例展示了不同复杂度和场景下的分析方式：

| 场景 | 深度 | 特点 |
|------|------|------|
| 工具选择 | 快速 | 2 个方案，简洁对比 |
| 功能实现 | 标准 | 3-4 个方案，详细评估 |
| 架构决策 | 深度 | 4-5 个方案，完整分析 |
| 快速验证 | 快速 | 聚焦验证方法 |
| 渐进式方案 | 标准 | 分阶段规划 |

根据实际场景选择合适的分析深度，避免过度分析或分析不足。
