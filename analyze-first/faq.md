## 基本问题

### Q: 什么时候应该使用 analyze-first？

**A**: 当你面临以下情况时：

- 功能实现有多种方式，不确定哪种最好
- 架构决策需要权衡多种因素
- 想要避免返工，提前规划
- 需要团队讨论和决策

**不适合的情况**：

- 问题已经很明确，不需要分析
- 时间紧迫，需要立即行动
- 问题太小，不值得分析
- 需要创意探索（用 brainstorming）

### Q: analyze-first 和 /plan、/spec 有什么区别？

**A**: 

| 工具 | 用途 | 场景 |
|------|------|------|
| **analyze-first** | 方案选择和决策 | "怎么实现 X" |
| **/plan** | 详细执行计划 | 选定方案后的步骤 |
| **/spec** | 完整规格说明 | 需求文档化 |

**流程**: analyze-first → 选择方案 → /plan → /spec

### Q: 分析需要多长时间？

**A**: 取决于分析深度：

| 深度 | 时间 | 适用场景 |
|------|------|----------|
| **快速** | ~30秒 | 简单工具选择 |
| **标准** | ~2分钟 | 功能实现决策 |
| **深度** | ~5分钟 | 架构决策 |

用户可以手动指定深度："快速分析一下" / "详细分析"

### Q: 分析结果准确吗？

**A**: 分析基于以下信息：

- 代码库上下文（技术栈、已有实现）
- 行业最佳实践
- 常见权衡因素

**局限性**：

- 没有实际运行代码，性能数据是估算的
- 没有考虑团队成员的技术背景
- 没有考虑部署环境的限制

建议：在全面实施前，先做 POC 验证

## 分析过程问题

### Q: AI 没有分析我的代码库怎么办？

**A**: 可能的原因：

1. **问题太简单**: 快速分析可能跳过代码库分析
2. **代码库太大**: AI 可能只分析相关部分
3. **权限问题**: AI 无法访问某些文件

**解决方法**：

- 明确要求："先分析一下我的代码库"
- 提供上下文："我用的是 React 18 + TypeScript"
- 指定文件："看看 src/components 目录"

### Q: 分析结果太泛泛怎么办？

**A**: 可能的原因：

1. **问题描述太模糊**: "我想优化性能"
2. **缺少约束条件**: 没有时间、资源限制
3. **没有指定技术栈**: AI 不知道用什么框架

**解决方法**：

- 提供具体信息："我想优化 React 应用的首屏加载时间，目前是 3 秒"
- 指定约束："时间 1 周，不能引入新依赖"
- 说明背景："我们用 Next.js 14，部署在 Vercel"

### Q: 分析结果与我的预期不符怎么办？

**A**: 可能的原因：

1. **优先级不同**: 你看重的方面和 AI 不同
2. **信息不足**: AI 没有完整了解你的情况
3. **经验差异**: AI 的经验和你的经验不同

**解决方法**：

- 明确优先级："我最看重开发速度，其次是可维护性"
- 提供更多信息："我们团队对 Redux 很熟悉"
- 追问原因："为什么推荐这个方案？"

### Q: 分析过程太慢怎么办？

**A**: 可能的原因：

1. **分析深度太深**: 深度分析需要更多时间
2. **问题太复杂**: 复杂问题需要更多分析
3. **代码库太大**: 分析代码库需要时间

**解决方法**：

- 指定深度："快速分析一下"
- 简化问题："先分析核心功能"
- 分阶段："先分析第一阶段"

## 方案选择问题

### Q: 所有方案都不满意怎么办？

**A**: 可能的原因：

1. **约束条件太严格**: 时间、资源限制太紧
2. **问题范围太大**: 需要拆分成更小的问题
3. **需求不明确**: 需要重新定义问题

**解决方法**：

- 调整约束："时间可以延长到 2 周"
- 拆分问题："先分析核心功能"
- 重新定义："我想实现的是..."
- 混合方案："能不能组合方案 A 和 B？"

### Q: 推荐的方案不适合我的情况怎么办？

**A**: 可能的原因：

1. **优先级不同**: AI 的推荐基于通用优先级
2. **特殊情况**: 你的情况有特殊约束
3. **经验差异**: AI 的经验和你的经验不同

**解决方法**：

- 说明优先级："我最看重的是..."
- 说明特殊情况："我们有这个约束..."
- 追问原因："为什么推荐这个方案？"
- 选择备选："我想用方案 B"

### Q: 如何在多个方案中选择？

**A**: 考虑以下因素：

1. **复杂度**: 你能接受多复杂的方案？
2. **工作量**: 你有多少时间？
3. **风险**: 你能承受多大风险？
4. **长期影响**: 这个方案长期来看如何？
5. **团队熟悉度**: 团队对这个方案熟悉吗？

**决策框架**：

- 如果时间紧迫 → 选择最简单的方案
- 如果质量优先 → 选择最稳健的方案
- 如果长期考虑 → 选择可维护性最好的方案
- 如果团队不熟悉 → 选择学习成本最低的方案

### Q: 混合方案怎么实现？

**A**: 混合方案是组合多个方案的优点：

**示例**：
```
方案 A: 简单但功能有限
方案 B: 复杂但功能完整

混合方案: 先用方案 A 实现核心功能，后续用方案 B 扩展
```

**实现步骤**：

1. 确定核心功能和扩展功能
2. 用简单方案实现核心功能
3. 预留扩展接口
4. 后续用更完整的方案扩展

### Q: 渐进式方案怎么规划？

**A**: 渐进式方案是分阶段实施：

**示例**：
```
阶段 1: 快速实现核心功能（1-2天）
阶段 2: 完善功能和优化（3-5天）
阶段 3: 高级功能和架构优化（1-2周）
```

**规划要点**：

- 每个阶段独立可验证
- 每个阶段有明确的交付物
- 每个阶段有明确的验证指标
- 阶段之间有明确的依赖关系

## 实施问题

### Q: 选择方案后怎么开始实施？

**A**: 建议的流程：

1. **可行性验证**: 先做 POC 或试点
2. **详细规划**: 使用 /plan 生成详细计划
3. **规格说明**: 使用 /spec 生成规格文档
4. **开始实施**: 按计划实施

**示例**：
```
用户: "我想用 NextAuth.js 实现用户认证"
AI: "建议先做 POC 验证，然后生成详细计划"
用户: "好的，先做 POC"
AI: [生成 POC 计划]
```

### Q: 实施过程中遇到问题怎么办？

**A**: 可能的原因：

1. **方案不适合**: 原来选的方案不适合实际情况
2. **技术难点**: 遇到了预期之外的技术难点
3. **需求变化**: 需求发生了变化

**解决方法**：

- 重新分析："这个方案遇到了问题，帮我重新分析"
- 调整方案："能不能用其他方式实现？"
- 拆分问题："先解决这个技术难点"

### Q: 如何验证方案是否可行？

**A**: 验证方式：

1. **快速 POC**: 最小化验证核心假设
2. **试点实现**: 先在一个模块应用
3. **技术调研**: 验证技术可行性
4. **基准测试**: 测量性能影响

**验证指标**：

- 功能是否正常
- 性能是否达标
- 开发体验如何
- 维护成本如何

### Q: 实施结果不符合预期怎么办？

**A**: 可能的原因：

1. **方案选择错误**: 原来选的方案不适合
2. **实施方式不对**: 实施方法有问题
3. **预期不合理**: 预期本身不合理

**解决方法**：

- 分析原因："为什么不符合预期？"
- 调整方案："要不要换个方案？"
- 调整预期："这个结果其实已经不错了"
- 记录经验："下次怎么避免这个问题？"

## 特殊情况

### Q: 问题太复杂，不知道从哪里开始？

**A**: 建议的流程：

1. **拆分问题**: 把大问题拆分成小问题
2. **确定优先级**: 先解决最重要的问题
3. **分阶段分析**: 每次分析一个小问题
4. **逐步推进**: 解决一个问题后再解决下一个

**示例**：
```
用户: "我想重构整个应用"
AI: "建议分阶段进行：
1. 先确定重构目标和优先级
2. 选择一个模块作为试点
3. 验证重构方法的有效性
4. 逐步扩展到其他模块"
```

### Q: 需要推翻之前的决策怎么办？

**A**: 建议的流程：

1. **分析原因**: 为什么需要推翻？
2. **评估影响**: 推翻会有什么影响？
3. **重新分析**: 基于新信息重新分析
4. **记录更新**: 更新决策记录

**示例**：
```
用户: "我觉得之前选的方案不合适"
AI: "可以重新分析。请告诉我：
1. 之前方案的哪些方面不符合预期？
2. 现在的约束条件有什么变化？
3. 有没有新的需求或限制？"
```

### Q: 团队意见不一致怎么办？

**A**: 建议的流程：

1. **收集意见**: 了解每个人的意见和理由
2. **分析权衡**: 分析每种意见的优缺点
3. **寻找共识**: 寻找大家都能接受的方案
4. **记录决策**: 记录最终决策和理由

**决策方法**：

- **投票**: 多数决定
- **讨论**: 充分讨论后决定
- **负责人决定**: 由负责人最终决定
- **试点验证**: 先试点，根据结果决定

### Q: 时间紧迫，来不及分析怎么办？

**A**: 建议的流程：

1. **快速分析**: 使用快速深度分析
2. **选择最简方案**: 选择最简单的方案
3. **先实现核心**: 只实现核心功能
4. **后续优化**: 后续再优化和完善

**示例**：
```
用户: "时间紧迫，明天就要上线"
AI: "建议使用快速分析，选择最简单的方案，先实现核心功能"
```

### Q: 技术选型涉及多个团队怎么办？

**A**: 建议的流程：

1. **收集需求**: 了解每个团队的需求和约束
2. **统一标准**: 制定统一的技术选型标准
3. **共同分析**: 让每个团队参与分析过程
4. **达成共识**: 让每个团队都同意最终方案

**协调要点**：

- 明确每个团队的职责
- 明确技术选型的标准
- 明确沟通和决策流程
- 明确实施和验收标准

## 总结

常见问题的核心解决思路：

1. **提供足够信息**: 问题描述越具体，分析越准确
2. **明确优先级**: 明确你最看重的方面
3. **验证假设**: 在全面实施前，先做 POC 验证
4. **记录决策**: 记录决策过程和理由
5. **持续改进**: 根据反馈优化分析方法
