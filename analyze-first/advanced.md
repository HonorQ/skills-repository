## 定制化选项

### 用户偏好设置

用户可以通过以下方式定制分析行为：

#### 1. 方案类型偏好

```
"只给我简单方案" / "only simple solutions"
→ 只推荐低复杂度方案，忽略复杂方案

"给我所有方案，包括复杂的"
→ 包含所有可行方案，无论复杂度
```

#### 2. 约束条件

```
"不要引入新依赖" / "no new dependencies"
→ 只推荐不引入新依赖的方案

"时间紧迫，给我最快的方案"
→ 优先推荐工作量最小的方案

"这是长期项目，考虑可维护性"
→ 优先推荐长期可维护的方案
```

#### 3. 分析深度

```
"快速分析一下" / "quick analysis"
→ 快速深度：2-3 个方案，简洁对比

"详细分析" / "deep dive"
→ 深度分析：4-5 个方案，完整评估
```

#### 4. 输出格式

```
"用表格对比" / "table format"
→ 使用表格展示方案对比

"给我详细报告"
→ 完整的分析报告

"简洁一点"
→ 精简输出，只保留关键信息
```

#### 5. 风险偏好

```
"避免高风险方案" / "avoid high-risk approaches"
→ 只推荐低风险方案

"可以接受中等风险"
→ 包含中等风险方案

"这是实验性项目，可以冒险"
→ 包含高风险高回报方案
```

#### 6. 验证方式

```
"先做 POC" / "always suggest POC first"
→ 优先建议 POC 验证

"直接试点实现"
→ 优先建议模块试点

"先调研技术可行性"
→ 优先建议技术调研
```

### 组合使用

可以组合多个偏好：

```
"快速分析，不要新依赖，避免高风险"
→ 快速深度 + 无新依赖 + 低风险

"详细分析，考虑长期可维护性，先做 POC"
→ 深度分析 + 长期优先 + POC 验证
```

## 团队协作

### 分享分析结果

#### 1. 导出为文档

分析结果可以导出为 Markdown 文档，方便团队分享：

```markdown
# 技术方案分析: [问题描述]

**日期**: YYYY-MM-DD
**分析者**: AI
**决策者**: [团队成员]

## 背景
[问题背景和约束]

## 方案对比
[方案对比表格]

## 推荐方案
[推荐方案和理由]

## 决策记录
[最终决策和理由]
```

#### 2. 团队评审

建议的评审流程：

1. **AI 分析**: 生成方案对比和推荐
2. **团队评审**: 团队成员讨论方案
3. **决策确认**: 确认最终方案
4. **记录存档**: 保存决策记录

#### 3. 异步协作

对于远程团队：

1. **AI 生成分析报告**
2. **分享到团队频道**（Slack/Teams/邮件）
3. **团队成员异步评审**
4. **收集反馈并更新分析**
5. **最终决策确认**

### 决策记录管理

#### 存储位置

建议将决策记录存储在项目中：

```
project/
├── docs/
│   ├── decisions/

│   │   ├── 2026-05-30-state-management.md
```

## 与其他 Skill 集成

### writing-plans

**触发时机**: 用户选择方案后

**流程**:
1. 用户选择方案
2. AI 调用 `writing-plans` skill
3. 生成详细执行计划

**示例**:
```
用户: "我想给应用添加用户认证功能"
AI: [分析方案，推荐 NextAuth.js]
用户: "1"  # 选择推荐方案
AI: [调用 writing-plans，生成详细执行计划]
```

### test-driven-development

**触发时机**: 方案涉及复杂逻辑或关键功能

**流程**:
1. 用户选择方案
2. AI 建议使用 TDD
3. 先写测试，再实现功能

**示例**:
```
用户: "我想实现一个复杂的状态管理逻辑"
AI: [分析方案，推荐 Zustand]
用户: "1"
AI: "建议使用 TDD，先写测试用例，再实现功能"
```

### code-review

**触发时机**: 功能实现完成后

**流程**:
1. 用户完成实现
2. AI 调用 `code-review` skill
3. 审查代码质量和最佳实践

**示例**:
```
用户: "我完成了用户认证功能的实现"
AI: [调用 code-review，审查代码]
```

### karpathy-guidelines

**触发时机**: 分析过程中，应用简洁性原则

**流程**:
1. AI 分析方案时
2. 应用 karpathy-guidelines 的简洁性原则
3. 优先推荐简单方案

**示例**:
```
用户: "我想优化代码结构"
AI: [应用简洁性原则，推荐最简单的方案]
```

## 决策记录管理

### 版本控制

将决策记录纳入版本控制：

```bash
# 添加决策记录
git add docs/decisions/

# 提交决策
git commit -m "docs: add decision record for auth solution"

# 查看决策历史
git log --oneline docs/decisions/
```

### 决策索引

创建决策索引文件：

```markdown
# 决策索引

| 日期 | 问题 | 决策 | 状态 |
|------|------|------|------|
| 2026-05-29 | 用户认证方案 | NextAuth.js | 已决策 |
| 2026-05-30 | 状态管理方案 | Zustand | 已决策 |
| 2026-05-31 | API 架构 | 保持 REST | 待决策 |
```

### 决策回顾

定期回顾决策记录：

1. **每月回顾**: 检查决策执行情况
2. **季度总结**: 分析决策效果
3. **年度复盘**: 总结经验教训

### 决策更新

当情况变化时，更新决策记录：

```markdown
## 决策更新

**更新日期**: YYYY-MM-DD
**更新原因**: [原因描述]
**新决策**: [新方案]
**新理由**: [为什么改变决策]
```

## 分析深度控制

### 自动深度判断

AI 根据问题特征自动判断深度：

| 特征 | 深度 | 原因 |
|------|------|------|
| 简单工具选择 | 快速 | 选项明确，影响小 |
| 功能实现 | 标准 | 需要权衡多种因素 |
| 架构决策 | 深度 | 影响长期，风险高 |
| 性能优化 | 标准/深度 | 取决于优化范围 |
| 重构 | 深度 | 影响广泛，需要谨慎 |

### 手动深度控制

用户可以手动指定深度：

```
"快速分析一下" → 快速深度
"详细分析" → 深度分析
"deep dive" → 深度分析
"just give me a quick answer" → 快速深度
```

### 深度调整建议

AI 可以建议调整深度：

```
AI: "这个问题涉及架构决策，建议使用深度分析。是否需要详细分析？"
用户: "是的，详细分析一下"
AI: [切换到深度分析]
```

## 特殊情况处理

### 方案都不满意

如果用户对所有方案都不满意：

```
用户: "这些方案都不太合适"
AI: "可以尝试以下方式：
1. 调整约束条件（如时间、资源）
2. 重新定义问题范围
3. 考虑混合方案
4. 寻找外部参考

是否需要重新分析？"
```

### 分析结果与预期不符

如果分析结果与用户预期不符：

```
用户: "我没想到会推荐这个方案"
AI: "让我解释一下推荐理由：
1. [理由1]
2. [理由2]
3. [理由3]

如果您的优先级不同，我可以调整推荐。您最看重哪些方面？"
```

### 需要推翻之前的决策

如果需要推翻之前的决策：

```
用户: "我觉得之前选的方案不合适"
AI: "可以重新分析。请告诉我：
1. 之前方案的哪些方面不符合预期？
2. 现在的约束条件有什么变化？
3. 有没有新的需求或限制？

我将基于新的信息重新分析。"
```

### 分析范围太大

如果分析范围太大：

```
用户: "我想重构整个应用"
AI: "这个范围太大，建议分阶段进行：
1. 先确定重构目标和优先级
2. 选择一个模块作为试点
3. 验证重构方法的有效性
4. 逐步扩展到其他模块

是否需要先分析试点模块？"
```

### 缺少关键信息

如果缺少关键信息：

```
AI: "为了给出更准确的分析，我需要了解：
1. 项目的技术栈是什么？
2. 有没有时间或资源限制？
3. 有没有团队偏好或经验限制？

请提供这些信息，我可以给出更针对性的建议。"
```

## 最佳实践

### 1. 明确问题范围

在分析前，明确问题范围和约束：

```
✅ "我想给 Next.js 应用添加用户认证，支持 OAuth，时间 2 周"
❌ "我想添加认证功能"
```

### 2. 提供足够上下文

提供项目背景和技术栈信息：

```
✅ "我们用 React 18 + TypeScript + Redux，有 30+ 组件"
❌ "我们用 React"
```

### 3. 明确优先级

明确你的优先级：

```
✅ "我最看重开发速度，其次是可维护性"
❌ "给我最好的方案"
```

### 4. 验证假设

在全面实施前，验证关键假设：

```
✅ "先做个 POC，验证技术可行性"
❌ "直接开始实现"
```

### 5. 记录决策

记录决策过程和理由：

```
✅ "我们选择方案 A，因为..."
❌ "我们选了方案 A"
```

### 6. 定期回顾

定期回顾决策效果：

```
✅ "上个月的决策执行得怎么样？"
❌ "做完就不管了"
```

## 总结

通过定制化选项、团队协作、Skill 集成和决策管理，analyze-first 可以适应各种复杂场景。关键是：

1. **明确需求**: 提供足够上下文和约束
2. **选择合适深度**: 根据问题复杂度选择分析深度
3. **团队协作**: 让团队参与决策过程
4. **记录决策**: 保存决策记录，方便回顾
5. **持续改进**: 根据反馈优化分析方法
