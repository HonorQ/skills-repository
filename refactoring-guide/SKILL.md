# Refactoring Guide

指导代码重构，保持代码整洁、可读、可维护。遵循"小步快跑、安全重构"的原则。

## 核心原则

### 1. 重构前先理解
不盲目改动，先确保理解代码意图：

```
✅ 先问：这段代码现在做什么？
✅ 再问：为什么要这样设计？
✅ 后想：哪里可以改进？
```

### 2. 小步快跑
每次只做一个小改动，确保可回滚：

```
❌ 错误：一次性重构整个模块
✅ 正确：提取一个函数 → 测试通过 → 提取下一个函数
```

### 3. 安全网优先
重构前确保有安全网：

```
- 有测试：先运行测试，确保全绿
- 无测试：先补充关键测试，再重构
- 复杂逻辑：先添加日志，确认行为一致
```

### 4. 功能不变
重构只改结构，不改行为：

```
❌ 禁止：顺便修复 Bug、添加功能
✅ 允许：改完后发现 Bug，另提修复任务
```

## 何时重构

### ✅ 应该重构的信号

| 信号 | 说明 | 优先级 |
|------|------|--------|
| 文件超过 300 行 | 难以快速理解 | 高 |
| 函数超过 50 行 | 职责过多 | 高 |
| 重复代码 | 违反 DRY 原则 | 高 |
| 命名不清晰 | 难以理解意图 | 中 |
| 嵌套层级过深 | 逻辑复杂 | 中 |
| 参数过多 | 函数耦合度高 | 中 |
| 注释解释代码 | 代码本身不清晰 | 低 |
| 魔法数字/字符串 | 可读性差 | 低 |

### ❌ 暂缓重构的情况

- 代码即将废弃或替换
- 没有测试且风险极高
-  deadline 紧迫，重构风险大
- 不理解业务逻辑，可能破坏功能

## 重构流程

### Step 0: 准备工作

```
1. 确保代码在版本控制中
2. 运行现有测试，确保全绿
3. 理解代码的业务逻辑
4. 确定重构范围（不要扩大化）
```

### Step 1: 识别坏味道

使用以下清单检查代码：

```
📋 代码坏味道检查清单

□ 重复代码（Duplicated Code）
  → 提取公共函数/类

□ 过长函数（Long Method）
  → 提取函数，按职责拆分

□ 过大类（Large Class）
  → 拆分类，提取职责

□ 过长参数列表（Long Parameter List）
  → 封装对象、使用 Builder 模式

□ 发散式变化（Divergent Change）
  → 一个类因不同原因修改 → 拆分

□ 霰弹式修改（Shotgun Surgery）
  → 改一处要改很多文件 → 合并相关功能

□ 依恋情结（Feature Envy）
  → 函数用了太多其他类的数据 → 搬移函数

□ 数据泥团（Data Clumps）
  → 总是一起出现的数据 → 封装成对象

□ 基本类型偏执（Primitive Obsession）
  → 用字符串/数字表示概念 → 封装成类

□ switch 语句（Switch Statements）
  → 考虑多态替代

□ 平行继承体系（Parallel Inheritance）
  → 两个继承体系一起变 → 合并

□ 冗赘类（Lazy Class）
  → 没用的类 → 内联或删除

□ 夸夸其谈未来性（Speculative Generality）
  → 过度设计 → 删除无用抽象

□ 临时字段（Temporary Field）
  → 对象某些时候不需要某些字段 → 提取类

□ 消息链（Message Chains）
  → a.b.c.d() → 隐藏委托关系

□ 中间人（Middle Man）
  → 过度委托 → 移除中间层

□ 狎昵关系（Inappropriate Intimacy）
  → 类之间过于了解 → 解耦

□ 异曲同工的类（Alternative Classes）
  → 接口相同实现不同 → 统一接口

□ 不完美的库类（Incomplete Library）
  → 引入外部类补充

□ 数据类（Data Class）
  → 只有字段没有行为 → 封装行为

□ 被拒绝的遗赠（Refused Bequest）
  → 子类不需要父类的方法 → 调整继承

□ 注释过多（Comments）
  → 用代码表达意图，删除注释
```

### Step 2: 制定重构计划

按优先级排序重构项：

```
## 重构计划

### 阶段 1：安全重构（低风险）
- [ ] 重命名变量/函数（语义清晰）
- [ ] 提取常量（魔法数字）
- [ ] 格式化代码

### 阶段 2：结构优化（中风险）
- [ ] 提取函数（过长函数）
- [ ] 内联临时变量
- [ ] 搬移函数

### 阶段 3：架构调整（高风险）
- [ ] 拆分类
- [ ] 调整继承关系
- [ ] 引入设计模式
```

### Step 3: 执行重构

#### 重构手法速查

**提取函数（Extract Function）**
```
适用：函数过长、代码块需要注释解释
步骤：
1. 创建新函数，根据意图命名
2. 复制代码到新函数
3. 检查变量，参数传入或内部声明
4. 替换原代码为函数调用
5. 测试
```

**内联函数（Inline Function）**
```
适用：函数代码比名字更清晰、过度委托
步骤：
1. 检查多态性
2. 找到所有调用点
3. 替换为函数体
4. 删除原函数
5. 测试
```

**提取变量（Extract Variable）**
```
适用：表达式复杂难以理解
步骤：
1. 将表达式赋值给临时变量
2. 用变量替换原表达式
3. 测试
```

**内联变量（Inline Variable）**
```
适用：变量名不比表达式更清晰
步骤：
1. 检查变量是否只赋值一次
2. 替换所有引用为表达式
3. 删除变量声明
4. 测试
```

**改变函数声明（Change Function Declaration）**
```
适用：函数名不清晰、参数设计不合理
步骤：
1. 如果需要，先分离查询和修改
2. 使用迁移式做法：
   - 创建新函数
   - 旧函数调用新函数
   - 逐步替换调用点
   - 删除旧函数
3. 测试
```

**封装变量（Encapsulate Variable）**
```
适用：public 字段需要控制访问
步骤：
1. 创建访问函数
2. 修改取值处为调用访问函数
3. 修改赋值处为调用设值函数
4. 限制字段访问权限
5. 测试
```

**变量改名（Rename Variable）**
```
适用：变量名不清晰
步骤：
1. 如果变量被广泛使用，使用封装变量
2. 改名并更新所有引用
3. 测试
```

**引入参数对象（Introduce Parameter Object）**
```
适用：一组参数总是一起传递
步骤：
1. 创建新类包含这些参数
2. 修改函数签名使用新类
3. 更新所有调用点
4. 测试
```

**函数组合成类（Combine Functions into Class）**
```
适用：一组函数操作同一组数据
步骤：
1. 创建新类
2. 将数据作为字段
3. 将函数搬移到类中
4. 更新调用点
5. 测试
```

**拆分阶段（Split Phase）**
```
适用：一个函数做了多件事
步骤：
1. 将第二阶段的代码提取为函数
2. 测试
3. 引入中间数据结构
4. 逐段提取并测试
```

### Step 4: 验证重构

```
✅ 验证清单

□ 功能一致性
  - 所有测试通过
  - 手动测试关键路径
  - 对比输入输出一致

□ 代码质量
  - 代码更清晰了
  - 重复减少了
  - 函数/类职责单一

□ 无回归
  - 没有新警告
  - 性能没有下降
  - 没有破坏接口契约
```

### Step 5: 提交与文档

```
📝 提交信息示例

refactor: 提取用户验证逻辑

- 将散落在 3 个地方的验证代码提取到 UserValidator 类
- 统一验证规则，消除重复
- 添加单元测试覆盖验证逻辑

相关任务: #123
```

## 常见重构场景

### 场景 1：函数过长

```
Before:
function processOrder(order) {
  // 50+ 行代码，包含验证、计算、保存、通知
}

重构步骤：
1. 提取 validateOrder(order)
2. 提取 calculateTotal(order)
3. 提取 saveOrder(order)
4. 提取 notifyUser(order)
5. processOrder 变成协调者
```

### 场景 2：重复代码

```
Before:
// 文件 A
function calculatePriceA(item) {
  let base = item.price;
  if (item.category === 'book') base *= 0.9;
  return base;
}

// 文件 B
function calculatePriceB(item) {
  let base = item.price;
  if (item.category === 'book') base *= 0.9;
  return base + item.shipping;
}

重构步骤：
1. 提取公共部分到 PriceCalculator 类
2. 提取 calculateBasePrice(item)
3. A 和 B 都调用公共方法
4. B 额外加上 shipping
```

### 场景 3：类过大

```
Before:
class UserManager {
  // 300+ 行，包含用户 CRUD、权限、统计、导出...
}

重构步骤：
1. 提取 UserRepository（数据访问）
2. 提取 PermissionService（权限）
3. 提取 UserStatistics（统计）
4. 提取 UserExporter（导出）
5. UserManager 变成门面类
```

### 场景 4：嵌套过深

```
Before:
if (user) {
  if (user.isActive) {
    if (user.hasPermission) {
      // 实际逻辑
    }
  }
}

重构步骤：
1. 使用卫语句提前返回
2. 或提取验证函数

After:
if (!user) return;
if (!user.isActive) return;
if (!user.hasPermission) return;
// 实际逻辑
```

## 重构安全技巧

### 1. 小步提交
```
每完成一个小重构就提交：
- 重命名变量 → 提交
- 提取函数 → 提交
- 搬移函数 → 提交
```

### 2. 特性开关
```
大重构时使用特性开关：
- 保留旧实现
- 新实现放在开关后
- 逐步切换流量
- 确认稳定后删除旧代码
```

### 3. 并行实现
```
接口重构时：
- 创建新接口
- 旧接口调用新接口
- 逐步迁移调用方
- 删除旧接口
```

### 4. 自动化检查
```
使用工具检查：
- Lint 检查代码风格
- 类型检查确保类型安全
- 测试覆盖率不下降
```

## 最佳实践

1. **不要同时重构和改功能**：只做一件事
2. **保持测试通过**：红-绿-重构循环
3. **代码审查**：重构后请他人 review
4. **记录决策**：复杂重构记录为什么这样改
5. **回滚准备**：保留回滚能力直到验证完成
6. **渐进式重构**：不要追求一步到位
7. **团队共识**：确保团队理解并接受新结构
